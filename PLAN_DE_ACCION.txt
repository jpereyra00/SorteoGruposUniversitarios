VISION ACTUALIZADA
------------------
Transformar la API en una plataforma educativa genérica con:
- Soporte multi-institución (desacoplado de UADE)
- Múltiples instancias de examen con historial
- Banco de preguntas y generación aleatoria
- Corrección asistida por IA (Spring AI)
- Métricas académicas completas
- Arquitectura hexagonal (ports & adapters) en el núcleo
- Servicios con responsabilidades claras, algunos independizables como microservicios

SECCIÓN A — DECISIONES DE ARQUITECTURA
================================================================================

A.1 — POR QUÉ HEXAGONAL Y NO SOLO CAPAS
-----------------------------------------
La arquitectura actual (Controller → Service → Repository) es frágil para
escalar: la lógica de negocio está acoplada a JPA, Spring, y los detalles
HTTP. La arquitectura hexagonal (ports & adapters) separa:

  DOMINIO (núcleo):
    - Entidades de negocio puras (sin anotaciones JPA)
    - Casos de uso / servicios de dominio
    - Puertos de entrada (lo que el mundo exterior puede pedirle al dominio)
    - Puertos de salida (lo que el dominio necesita del mundo exterior)

  ADAPTADORES DE ENTRADA (driving):
    - REST Controllers (HTTP → caso de uso)
    - Importación CSV/JSON

  ADAPTADORES DE SALIDA (driven):
    - Repositorios JPA (caso de uso → base de datos)
    - Clientes Spring AI (caso de uso → LLM)
    - Generador PDF (caso de uso → archivo binario)
    - Mailer (caso de uso → SMTP, futuro)

  Beneficio concreto: si mañana cambiás de H2 a PostgreSQL, de OpenAI a
  Ollama, o de REST a gRPC, solo tocás el adaptador. El dominio no se mueve.

A.2 — QUÉ QUEDA COMO MONOLITO MODULAR Y QUÉ SALE COMO MICROSERVICIO
---------------------------------------------------------------------

  MÓDULO / SERVICIO              DEPLOYMENT            JUSTIFICACIÓN
  ----------------------------   -------------------   --------------------------------
  exam-core                      Monolito modular      CRUD fuertemente acoplado entre
  (students, groups, exams,      (módulo Maven)        entidades, compartir transacción
  topics, institutions, grades)                        es correcto aquí

  pdf-service                    Microservicio         Stateless, sin BD propia,
                                 independiente         resource-intensive (CPU/mem),
                                                       se puede escalar y reemplazar
                                                       sin tocar el core

  question-bank-service          Microservicio         Dominio independiente, puede
                                 independiente         ser compartido por otras apps
                                                       o instituciones distintas

  ai-correction-service          Microservicio         Llama a APIs externas costosas,
                                 independiente         necesita su propio rate-limit,
                                                       fallback, y SLA distinto

  analytics-service              Microservicio         Read-heavy, event-driven,
                                 (consumidor de        sus datos se pueden recalcular
                                  eventos)             desde el core. No escribe en
                                                       el core.

  Patrón de comunicación:
  - exam-core → pdf-service:          REST síncrono (pide PDF, espera bytes)
  - exam-core → ai-correction:        REST asíncrono + polling, o eventos
  - exam-core → question-bank:        REST síncrono (pide preguntas)
  - exam-core → analytics-service:    Eventos de dominio (publica, no espera)
  - Todos → API Gateway:              Un solo punto de entrada para el frontend

FASE 1 — REESTRUCTURACIÓN A ARQUITECTURA HEXAGONAL (exam-core)

ITEM 1.1 — Convertir a proyecto Maven multi-módulo
----------------------------------------------------
  - Crear parent pom.xml con módulos: exam-core, pdf-service, question-bank-service,
    ai-correction-service, analytics-service
  - Mover código actual a exam-core/
  - Verificar que todo sigue compilando y funcionando

ITEM 1.2 — Crear capa domain/ con modelos puros
------------------------------------------------
  Para cada entidad actual, crear su equivalente de dominio:
  - domain/model/Student.java          ← POJO sin @Entity, sin Spring
  - domain/model/Exam.java
  - domain/model/ExamGroup.java
  - domain/model/Topic.java
  - domain/model/Institution.java      ← nuevo (Fase 2)
  - domain/model/Grade.java            ← nuevo (Fase 3)

  Regla: ninguna clase en domain/ puede importar Spring, JPA, Jackson, etc.

ITEM 1.3 — Definir puertos de entrada (use cases)
---------------------------------------------------
  Interfaces en domain/port/in/:
  - CreateStudentUseCase        { Student create(CreateStudentCommand cmd); }
  - ImportStudentsUseCase       { List<Student> importFromText(String raw); }
  - CreateExamUseCase           { Exam create(CreateExamCommand cmd); }
  - GenerateGroupsUseCase       { List<ExamGroup> generateRandom(GenerateGroupsCommand cmd); }
  - AssignTopicsUseCase         { void assign(AssignTopicsCommand cmd); }
  - RegisterGradeUseCase        { Grade register(RegisterGradeCommand cmd); }
  - ExportPdfUseCase            { byte[] export(Long groupId); }

  Command = record inmutable con los datos necesarios para ejecutar el caso de uso

ITEM 1.4 — Definir puertos de salida (dependencias del dominio)
----------------------------------------------------------------
  Interfaces en domain/port/out/:
  - LoadStudentPort             { Optional<Student> load(Long id); }
  - SaveStudentPort             { Student save(Student student); }
  - LoadExamPort                { Optional<Exam> load(Long id); }
  - SaveExamPort                { Exam save(Exam exam); }
  - GeneratePdfPort             { byte[] generate(ExamGroup group, Exam exam); }
  - LoadQuestionsPort           { List<Question> loadByTopic(Long topicId, int count); }
  - PublishDomainEventPort      { void publish(DomainEvent event); }

ITEM 1.5 — Mover lógica de negocio a domain/service/
------------------------------------------------------
  - GroupingDomainService:   lógica de distribución aleatoria/manual de grupos
  - GradingDomainService:    cálculo de nota final, asignación de StudentStatus
  - TopicAssignmentService:  lógica de asignación de temas (con/sin repetición)

  Estos servicios reciben objetos de dominio y devuelven objetos de dominio.
  No saben nada de HTTP ni de base de datos.

ITEM 1.6 — Implementar adaptadores de salida JPA
-------------------------------------------------
  En infrastructure/adapter/out/jpa/:
  - Mappers: domain/model ↔ @Entity JPA (MapStruct o manual)
  - JpaStudentRepository implements SaveStudentPort, LoadStudentPort
  - JpaExamRepository implements SaveExamPort, LoadExamPort
  - etc.

  Los @Entity JPA viven solo en infrastructure/, nunca en domain/.

ITEM 1.7 — Adaptar Controllers existentes
-------------------------------------------
  - Los Controllers pasan a llamar puertos de entrada (use cases)
  - Ya no llaman directamente a los Services actuales
  - Se eliminan los @Service actuales (su lógica migró a domain/service/)

FASE 6 — MICROSERVICIO: analytics-service (event-driven)
Estimación: 3-4 sesiones
================================================================================

CONTEXTO
--------
  Las métricas son read-heavy y no necesitan transacciones con el core.
  Se alimentan de eventos de dominio (GradeRegistered, ExamClosed) y
  calculan sus propias proyecciones. Si el servicio cae, se puede
  reconstruir desde los eventos almacenados.

ITEM 6.1 — Eventos de dominio en exam-core
--------------------------------------------
  domain/event/:
  - GradeRegisteredEvent(examId, studentId, finalGrade, status, timestamp)
  - ExamClosedEvent(examId, timestamp)
  - GroupGeneratedEvent(examId, groupCount, studentCount, timestamp)

  PublishDomainEventPort (interfaz de salida) → implementado con:
    a) Spring Events (in-process, suficiente si no hay bus externo)
    b) Kafka/RabbitMQ (si se quiere bus real entre servicios)

  Para empezar: usar Spring ApplicationEvents (in-process). Migrar a Kafka
  cuando haya necesidad real de desacoplamiento físico.

ITEM 6.2 — Crear módulo analytics-service
-------------------------------------------
  Estructura hexagonal:
  analytics-service/
  ├── domain/
  │   ├── model/
  │   │   ├── ExamSnapshot.java         ← proyección calculada de un examen
  │   │   ├── StudentProfile.java       ← historial agregado del alumno
  │   │   └── SubjectReport.java        ← reporte por materia
  │   └── service/
  │       └── MetricsCalculationService.java
  ├── application/
  │   └── usecase/
  │       ├── ComputeExamMetricsUseCase.java
  │       └── ComputeImprovementPotentialUseCase.java
  └── infrastructure/
      ├── adapter/in/
      │   ├── event/GradeRegisteredListener.java  ← consume eventos del core
      │   └── rest/MetricsController.java
      └── adapter/out/
          └── jpa/AnalyticsRepository.java        ← BD propia (H2/Postgres)

  El analytics-service tiene su propia base de datos. No comparte BD con exam-core.

ITEM 6.3 — Endpoints de métricas
----------------------------------
  GET /api/metrics/exams/{examId}
  → {
      totalStudents, graded, absent,
      promoted: { count, percentage, students[] },
      approved:  { count, percentage, students[] },
      atRisk:    { count, percentage, students[] },
      failed:    { count, percentage, students[] },
      averageGrade, highestGrade, lowestGrade,
      gradeDistribution: [ { range, count } ... ],
      topicDifficulty:   [ { topicName, avgGrade } ... ]
    }

  GET /api/metrics/students/{studentId}/history
  → { exams[], summary: { trend, avgGrade, totalExams } }

  GET /api/metrics/subjects?name=Cálculo I
  → { overallApprovalRate, examEvolution[] }

  GET /api/metrics/exams/{examId}/at-risk
  → lista de alumnos AT_RISK para intervención temprana

  GET /api/metrics/exams/{examId}/improvement-potential
  → {
      topicsToReinforce: [ { topic, impactScore, studentsImpacted } ],
      estimatedPassRateIncrease
    }

FASE 7 — MICROSERVICIO: question-bank-service
Estimación: 4-5 sesiones
================================================================================

CONTEXTO
--------
  El banco de preguntas es un dominio independiente que puede ser compartido

ITEM 7.1 — Modelo de dominio
------------------------------

  - Question(id, text, explanation, topicId, topicName, difficulty, type, points, active)
  - QuestionOption(id, questionId, text, isCorrect, explanation)
  - QuestionDifficulty enum: EASY, MEDIUM, HARD
  - QuestionType enum: OPEN, MULTIPLE_CHOICE, TRUE_FALSE
  - ExamTemplate(id, name, institutionId, totalPoints, rules[])
  - ExamTemplateRule(topicId, easyCount, mediumCount, hardCount, type)

ITEM 7.2 — CRUD banco de preguntas
------------------------------------
  Endpoints:
  GET    /api/questions?topicId=&difficulty=&type=&page=&size=
  POST   /api/questions
  PUT    /api/questions/{id}

  POST   /api/questions/import        ← import JSON/CSV masivo
  GET    /api/questions/export        ← export JSON completo del banco

ITEM 7.3 — ExamTemplate y generación aleatoria
------------------------------------------------
  GET/POST/PUT/DELETE /api/templates
  POST /api/templates/{id}/generate?examGroupId={id}
  → retorna lista de Question seleccionadas según las reglas del template

  Lógica de selección:
  - Para cada ExamTemplateRule: SELECT preguntas WHERE topic=X AND difficulty=Y
    ORDER BY RANDOM() LIMIT N
  - Si allowRepetition=false: excluir IDs ya usados en el mismo examen
  - Si no hay suficientes preguntas: BusinessException con mensaje descriptivo

ITEM 7.4 — Adaptador en exam-core
-----------------------------------

ITEM 7.5 — PDF con preguntas (delegar a pdf-service)
-----------------------------------------------------

FASE 8 — MICROSERVICIO: ai-correction-service
Estimación: 4-6 sesiones | Requiere modelo de IA configurado
================================================================================

CONTEXTO
--------
  La corrección con IA llama a APIs externas costosas (OpenAI) o modelos

ITEM 8.1 — Setup Spring AI
-----------------------------
  Dependencias:
  - spring-ai-bom (BOM)
  - spring-ai-openai-spring-boot-starter   ← para producción (gpt-4o-mini)
  - spring-ai-ollama-spring-boot-starter   ← para desarrollo local gratuito

  Configuración:
  application-dev.properties:
    spring.ai.ollama.base-url=http://localhost:11434
    spring.ai.ollama.chat.model=llama3.2

  application-prod.properties:
    spring.ai.openai.api-key=${OPENAI_API_KEY}
    spring.ai.openai.chat.model=gpt-4o-mini

ITEM 8.2 — Modelo de dominio
------------------------------
  domain/model/:
  - CorrectionRequest(questionText, expectedAnswer, studentAnswer, maxPoints)
  - CorrectionResult(suggestedScore, reasoning, commonMistake, confidence)
  - StudentAnswer(id, studentGradeId, questionId, answerText, answerImagePath, reviewed)
  - AiCorrection(id, studentAnswerId, suggestedScore, reasoning, commonMistake,
                  confidence, acceptedByTeacher, generatedAt)
  - InstructionalInsight(examId, topicName, errorPattern, recommendation,
                          affectedStudents, affectedPercentage)

ITEM 8.3 — Servicio de corrección
-----------------------------------
  Prompt para corrección de respuesta abierta:
  ┌─────────────────────────────────────────────────────────────────┐
  │  Sos un asistente académico. Evaluá la respuesta del alumno.   │
  │                                                                  │
  │  PREGUNTA: {questionText}                                       │
  │  RESPUESTA ESPERADA: {expectedAnswer}                           │
  │  RESPUESTA DEL ALUMNO: {studentAnswer}                          │
  │  PUNTAJE MÁXIMO: {maxPoints}                                    │
  │                                                                  │
  │  Devolvé SOLO un JSON:                                          │
  │  {                                                               │
  │    "suggestedScore": <0 a maxPoints>,                           │
  │    "reasoning": "<explicación breve>",                          │
  │    "commonMistake": "<patrón de error o null>",                 │
  │    "confidence": <0.0 a 1.0>                                    │
  │  }                                                               │
  └─────────────────────────────────────────────────────────────────┘

  Fallback: si la IA no responde en N segundos, guardar la respuesta como
  pendiente de revisión manual (reviewed=false, suggestedScore=null).

ITEM 8.4 — Análisis de patrones de error
------------------------------------------
  Prompt para análisis de patrones en un examen:
  ┌─────────────────────────────────────────────────────────────────┐
  │  Tenés los siguientes errores comunes en el tema "{topicName}": │
  │  {lista de commonMistake strings}                               │
  │                                                                  │
  │  Respondé SOLO un JSON:                                         │
  │  {                                                               │
  │    "errorPattern": "<descripción del patrón general>",          │
  │    "recommendation": "<qué debería reforzar el docente>",       │
  │    "affectedPercentage": <0 a 100>                              │
  │  }                                                               │
  └─────────────────────────────────────────────────────────────────┘

ITEM 8.5 — Endpoints
----------------------
  POST /api/answers/{studentAnswerId}/correct-ai    ← corregir con IA
  PUT  /api/answers/{studentAnswerId}/accept-ai     ← docente acepta

  GET  /api/exams/{examId}/common-mistakes          ← informe de errores
  GET  /api/exams/{examId}/ai-summary               ← cuántos usaron IA, confianza promedio
